    Tutorial

    Writing your first Aquilify app ¶

    Let’s learn by example.

    Throughout this tutorial, we’ll walk you through the creation of a basic Todo application.

    • A public site that lets people view todo and vote in them.

    We’ll assume you have Aquilify installed already. You can tell Aquilify is installed and which version by running the following command in a shell prompt (indicated by the $ prefix):

    $ aquilify --version

    If Aquilify is installed, you should see the version of your installation. If it isn’t, you’ll get an error telling “No module named aquilify.

    This tutorial is written for Aquilify 1.14, which supports Python 3.10 and later. If the Aquilify version doesn’t match, you can refer to the tutorial for your version of Aquilify by using the version switcher at the bottom right corner of this page, or update Aquilify to the newest version. If you’re using an older version of Python, check What Python version can I use with Aquilify? to find a compatible version of Aquilify.

    See How to install Aquilify for advice on how to remove older versions of Aquilify and install a newer one.

    Info

    Where to get help:

    If you’re having trouble going through this tutorial, please head over to the Getting Help section of the FAQ.

    Creating a project ¶

    If this is your first time using Aquilify, you’ll have to take care of some initial setup. Namely, you’ll need to auto-generate some code that establishes a Aquilify project – a collection of settings for an instance of Aquilify, Aquilify-specific options and application-specific settings.

    From the command line, cd into a directory where you’d like to store your code, then run the following command:

    $ aquilify create-app
    Then it ask to name you app in this case we naming todo then it ask for setup Electrus ( Optional ) either select YES or NO.

    Info

    Note

    You’ll need to avoid naming projects after built-in Python or Aquilify components. In particular, this means you should avoid using names like aquilify (which will conflict with Aquilify itself) or test (which conflicts with a built-in Python package).

    Success

    Where should this code live?

    If your background is in plain old PHP (with no use of modern frameworks), you’re probably used to putting code under the web server’s document root (in a place such as /var/www). With Aquilify, you don’t do that. It’s not a good idea to put any of this Python code within your web server’s document root, because it risks the possibility that people may be able to view your code over the web. That’s not good for security.

    Put your code in some directory outside of the document root, such as /home/mycode.

    Let’s look at what aquilify created:

    todo/
    app/
        __init__.py
        app.py
    lifespan/
        __init__.py
    __init__.py
    __root__.py
    asgi.py
    tools.py
    models.py
    settings.py
    routing.py
    views.py
    packlib.lxe
    config.cfg
    These files are:

    • The outer todo/ root directory is a container for your project. Its name doesn’t matter to Aquilify you can rename it to anything you like.

    • todo/app: A dynamic routing directory using Aquilify Schematic inside the todo/app/main.py some codes were auto generated for configuration of Schematic.

    • todo/init.py: An empty file that tells Python that this directory should be considered a Python package. If you’re a Python beginner, read more about packages in the official Python docs.

    • todo/root.py: Main pipline for your application flow it controls the entire application inside the root.py and variable defined called instance it's the main entry-point for you application to connect with web.

    • todo/settings.py: Settings/configuration for this Aquilify project. Aquilify settings will tell you all about how settings work.

    • todo/routing.py: The URL declarations for this Aquilify project; a “table of contents” of your Aquilify-powered site. You can read more about URLs in URL dispatcher.

    • todo/views.py: The views.py is file where you can define all you views.

    • todo/packlib.lxe: This file contain pipline releated to your project, if you made any further changes it may reflect to your project and maybe it broke the entire project.

    • todo/config.cfg: This file contain data releated to your APPLICATION & ASGI_SERVER there are two block do not make changes to the AQUILIFY block, and for ASGI_SERVER block by default it uses NETIX asgi web server you may configure it to either UVICORN or NETIX, support for more web server's soon...

    Success

    By default, the runserver command starts the development server on the internal IP at port 8000.

    If you want to change the server’s port, open the config.cfg and inside the ASGI_SERVER block change the port. For instance, this command starts the server on port 8000:

    [ASGI_SERVER]
port = 8080
    If you want to change the server’s IP, pass it along with the port. For example, to listen on all available public IPs (which is useful if you are running Vagrant or want to show off your work on other computers on the network), use:

    [ASGI_SERVER]
host = 0.0.0.0

    Success

    Automatic reloading of runserver

    The development server automatically reloads Python code for each request as needed. You don’t need to restart the server for code changes to take effect. However, some actions like adding files don’t trigger a restart, so you’ll have to restart the server in these cases.

    Write your first view ¶

    Let’s write the first view. Open the file todos/views.py and put the following Python code in it:

    # todo/views.py

async def index():
    return {"message": "Hello, world. You're at the todo index."}

    This is the simplest view possible in Aquilify. To call the view, we need to map it to a URL - and for this we need a URLconf.

    To create a URLconf in the todo directory, open a file called routing.py.

    In the todo/routing.py file include the following code:

    from aquilify.core.routing import rule

from . import views

ROUTER = [
    rule('/', endpoint=views.index, name='Index Route'),
]

    You have now wired an index view into the URLconf. Verify it’s working with the following command:

    $ aquilify runserver

    Go to http://localhost:8000/ in your browser, and you should see the text “Hello, world. You’re at the todo index.”, which you defined in the index view.

    Error

    Page not found?

    If you get an error page here, check that you’re going to http://localhost:8000/ not in any other route.

    The rule() function is passed six arguments, two required: path and endpoint, and other's are optional: kwargs, and name. At this point, it’s worth reviewing what these arguments are for.

    rule() argument: path

    path is a string that contains a URL pattern. When processing a request, Aquilify starts at the first pattern in ROUTING and makes its way down the list, comparing the requested URL against each pattern until it finds one that matches.

    rule() argument: endpoint ¶

    When Aquilify finds a matching pattern, it calls the specified view function with an Request object as the first argument and any “captured” values from the route as keyword arguments. We’ll give an example of this in a bit.

    Info

    Overview

    The application consists of Python code for handling CRUD operations (Create, Read, Update, Delete) for tasks. It includes functionalities to add, complete, and delete tasks.

    Task Management:

    • TaskSchema: Defines the schema for a task using Aquilify's serializer to validate task-related data such as the task name, status, and due date.
    • get_total_tasks_count(): Asynchronously retrieves the total count of tasks from the database. CRUD Operations:
    • index(request): Retrieves paginated tasks and prepares the context for rendering on the UI. It retrieves tasks based on pagination parameters (page number and items per page).
    • add_task(request): Adds a new task to the database. Validates and processes task data received from the form submission.
    • complete_task(task_id): Marks a specific task as completed in the database.
    • delete_task(task_id): Deletes a task from the database.

    Writing more views ¶

    Now let’s add a few more views to todo/views.py. These views are slightly different, because they take an argument:

    from typing import (
    Any, 
    Dict, 
    List, 
    Union
)

from aquilify.shortcuts import render
from aquilify.responses import JsonResponse

from . import db

async def get_total_tasks_count() -> int:
    return await db.todos_collection.count_documents({})

async def index(request: Any) -> JsonResponse:
    try:
        page_number: int = int(request.args.get('page', 1))
        items_per_page: int = int(request.args.get('limit', 10))

        total_tasks: int = await get_total_tasks_count()
        all_tasks: List[Dict[str, Any]] = list(await db.todos_collection.fetch_all())

        start_index: int = (page_number - 1) * items_per_page
        end_index: int = start_index + items_per_page
        paginated_tasks: List[Dict[str, Any]] = all_tasks[start_index:end_index]

        context: Dict[str, Union[List[Dict[str, Any]], Dict[str, Union[int, str]]]] = {
            "todos": paginated_tasks,
            "pagination": {
                "total_tasks": total_tasks,
                "page": page_number,
                "limit": items_per_page
            }
        }

        return await render(request, 'index.html', context)

    except (ValueError, KeyError) as e:
        return JsonResponse({'error': f'Invalid page or limit value provided: {e}'}, status=400)
    except Exception as e:
        return JsonResponse({'error': f'Failed to fetch todos: {e}'}, status=500)

    Let's add some more views

    • Creating a add_task view.

    from datetime import datetime
from typing import Dict, Any

from aquilify.responses import JsonResponse
from aquilify.core.backend.serializer import Schema, fields, FieldValidationError

from . import db

class TaskSchema(Schema):
    task: str = fields.String(required=True)
    status: str = fields.String()
    due_date: datetime = fields.Date()

async def add_task(request) -> JsonResponse:
    try:
        task_data: Dict[str, Any] = await TaskSchema().loads(await request.form())
        task_data['created_at']: str = datetime.now().strftime("%m/%d/%Y, %H:%M:%S")

        result: Any = await db.todos_collection.insert_one(task_data)
        if result:
            return JsonResponse({'message': 'Task added successfully!'})

    except FieldValidationError as e:
        return JsonResponse({'error': f'Failed to add task field: {e}'}, status=400)
    except Exception as e:
        return JsonResponse({'error': f'Failed to add task: {e}'}, status=500)

    Info

    The Schema class stands at the core of the serializer functionality, allowing seamless serialization and deserialization operations through its loads and dumps functions, accepting the data argument. Notably, it offers an array of built-in fields like String, Integer, and more, streamlining the process of defining data structures and facilitating serialization effortlessly.

    import asyncio
from aquilify.backend.serializer import Schema, fields, FieldValidationError

class Users(Schema):
    username = fields.String(required=True, error_messages={'required': "Username field is required!"})
    email = fields.Email(required=True, error_messages={"validation": 'Invalid email format!'})
    password = fields.Password(required=True, min_length=8, max_length=16, special_chars=True, error_messages={'validation': "Invalid password format!"})

async def main():
# Due to it's asynchnorous nature you must have to await it.
    data = {
        "username": "myusername",
        "email": "demo@gmail.com",
        "password": "demo@1234"
    }

    result = await Users().loads(data)
    print(result) # print the serialized data
    result = await Users().dumps(result)
    print(result) # print the deserialized data

if __name__ == "__main__":
    asyncio.run(main())

    Aquilify provides a inbuilt serializer & validator.

    TaskSchema is a set of rules defining the structure of a task. It ensures that a task has certain attributes (task, status, due_date) and their expected data types (str, datetime, etc.). fields.String() and fields.Date() define the expected types for each attribute.

    • Adding a Task:

      This code snippet handles adding a new task to a database. It receives a request and extracts data from the form submitted in that request. Validates this data against the TaskSchema rules to ensure it meets the required structure.

    • Processing and Saving the Task:

      After validating the task data, it adds a timestamp for the task creation (created_at).
The current timestamp is generated using datetime.now().strftime() to represent the current date and time in a specific format ("%m/%d/%Y, %H:%M:%S").
This timestamp is added to the task data.

    • Saving to the Database:

      Attempts to insert the validated and timestamped task data into a database (db.todos_collection.insert_one(task_data)).
If successful, it returns a success message as a JsonResponse ({'message': 'Task added successfully!'}).

    • Error Handling:

      If there are validation errors (e.g., task data doesn't match TaskSchema requirements), it catches a FieldValidationError and returns an error response with a 400 status code ({'error': 'Failed to add task field: {error_message}'}).
If any unexpected error occurs during the addition process, it catches it as a general exception and returns a server error response with a 500 status code ({'error': 'Failed to add task: {error_message}'}).

    Creating more views.

    from typing import Any

from aquilify.responses import JsonResponse

from . import db

async def complete_task(task_id: str) -> JsonResponse:
    try:
        result: Any = await db.todos_collection.update_one({'_id': task_id}, {'$set': {'status': 'Completed'}})

        if result:
            return JsonResponse({'message': 'Task marked as completed!'})
        else:
            return JsonResponse({'error': 'Task not found!'}, status=404)

    except Exception as e:
        return JsonResponse({'error': f'Error marking task as completed: {e}'}, status=500)

async def delete_task(task_id: str) -> JsonResponse:
    try:
        result: Any = await db.todos_collection.delete_one({'_id': task_id })

        if result:
            return JsonResponse({'message': 'Task deleted successfully!'})
        else:
            return JsonResponse({'error': 'Task not found!'}, status=404)

    except Exception as e:
        return JsonResponse({'error': f'Error deleting task: {e}'}, status=500)

    from aquilify.core.routing import rule

    Steps in complete_task:

    • Updating Task Status: Tries to change a task's status to 'Completed' in the database using the given task_id. Checks the result of this update.

    • Result Check: If the task was successfully updated, it sends a message saying the task was completed. If the task wasn't found, it informs that the task doesn't exist.

    • Error Handling: If something unexpected happens during this process, it reports an error.

    Steps in delete_task:

    • Deleting Task: Tries to remove the task from the database using the given task_id. Checks the result of this deletion.

    • Result Check: If the task was successfully deleted, it sends a message confirming the deletion. If the task wasn't found, it indicates that the task doesn't exist.

    • Error Handling: If something unexpected occurs during this process, it reports an error.
